# Feature Plan (Spec-Driven Development)

Feature development specifications are centralized here.
The specification is the Single Source of Truth; verbal decisions are not valid.

## Principle: Update Spec, Then Implement

1. When a specification change occurs, first update the documents in `docs/plan/<feature>/`
2. Only after the documents are updated should code changes be made
3. Verbal or chat agreements are not specifications. Only what is written down counts as specification

## Directory Structure

```
docs/plan/
├── README.md                    # This document
└── <feature-name>/              # Directory per feature
    ├── 00_OVERVIEW.md           # Feature overview, purpose, scope
    ├── 01_TYPES.md              # Type definitions, Zod schemas
    ├── 02_PLUGINS.md            # Platform plugin specifications
    ├── 03_CLIENT_API.md         # Public API surface
    ├── 04_INFRASTRUCTURE.md     # RestManager, rate limiting, auth, OTel
    ├── 05_PACKAGE_STRUCTURE.md  # Monorepo layout, build, packaging
    └── CHECKLIST.md             # Implementation checklist (phases and session notes)
```

Files (`.md`) auto-generated by Claude Code's plan mode are also saved in this directory.
Feature directories and auto-generated files use different naming conventions, so they do not conflict.

## Workflow

### Step 1: Specification Generation (`/plan-feature`)

Create structured specification documents from ambiguous requirements.

- Gather requirements through interviews and write specifications in the file structure above
- Specifications follow the SDK layer order: Types -> Infrastructure -> Plugins -> Client API
- Mark undecided items as `TBD` and record the next actions

### Step 2: Checklist Generation (`/init-impl`)

Read the specifications and generate a phased implementation checklist.

- Implementation order is bottom-up: Types -> Infrastructure -> Plugins -> Client API
- Each phase includes a goal, checklist, test commands, and session notes
- Session notes record Done / Next / Risks and TODO items

### Step 3: Phase Execution

Follow the checklist and implement phase by phase.

- Initialize session notes at the start of each phase
- Check off completed tasks
- If a specification change is needed, update the specification document first, then change the code

## How to Write Session Notes

Always record the following 3 items in the Session Notes of each phase.

```markdown
### Session Notes

YYYY-MM-DD

- Done: What was completed
- Next: What to do next
- Risks/TODO: Unresolved issues
```

Use these 3 items to restore context when resuming across days or sessions.
